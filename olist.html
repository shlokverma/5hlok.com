Retail Analytics Engineering

Modeling 100k+ e-commerce orders into a production-grade analytics layer

The problem

Retail companies like Aritzia, Lululemon, and Arc'teryx sit on enormous amounts of transactional data — but raw order tables don't answer business questions. An analyst shouldn't have to write a 40-line SQL query every time someone asks "what's our repeat purchase rate by product category?"

This project builds the analytics layer that sits between raw data and business decisions — clean, tested, documented models that make those questions trivial to answer.

The dataset

The Olist Brazilian E-Commerce dataset contains 100k+ real orders from 2016–2018, across customers, products, sellers, payments, and reviews. It's a realistic proxy for what a mid-size DTC brand's transactional database looks like.

Design decision

I considered modeling cart abandonment metrics but the dataset only contains transactional records — no clickstream or session data. In production this would require an additional source like Segment or Amplitude feeding a separate events table. Noting the limitation rather than faking it.

Data model

Star schema with two fact tables at different grains — one for order-level analysis (funnel, fulfillment, cancellations) and one for item-level analysis (product performance, revenue by category).

Schema diagram — coming Day 4
  • fct_orders — one row per order, order-level measures
  • fct_order_items — one row per order + product, item-level measures
  • dim_customers — customer attributes + cohort month
  • dim_products — product attributes + category
  • dim_dates — calendar table for time-based analysis
Design decision

Two fact tables rather than one because order-grain and item-grain answer fundamentally different questions. Collapsing them would either duplicate order-level measures or force awkward aggregations on every query.

Metrics this model answers

  • Repeat purchase rate by customer cohort
  • Customer lifetime value (LTV)
  • Revenue and order volume by product category
  • Cancellation rate (pre-fulfillment)
  • Average fulfillment time by seller / region
  • Review score distribution by product and category

Day 1
Environment setup + data ingestion
Set up Snowflake free trial, created raw schema, loaded 6 Olist CSVs. Hit a gotcha — Snowflake's Load Data wizard was silently failing due to inconsistent quoting in the CSVs. Diagnosed by checking copy history vs running a SELECT. Fixed by running COPY INTO with FIELD_OPTIONALLY_ENCLOSED_BY set. Also learned that CREATE OR REPLACE is destructive — was accidentally re-running DDL and wiping tables each time. Lesson: separate worksheets per purpose.

What I'd do differently in production

  • Use Fivetran or Airbyte for ingestion instead of manual COPY INTO
  • Add incremental models for large fact tables rather than full refreshes
  • Layer in a semantic layer (dbt metrics or Cube.js) so business users can self-serve
  • Add clickstream data source to enable cart abandonment and funnel analysis
